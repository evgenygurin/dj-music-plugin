# Tiered Audio Analysis — Design Report

> Как перестать тратить часы на анализ треков, которые никогда не попадут в сет

## Проблема

Сейчас: один вызов `analyze_track` прогоняет все 7 анализаторов и пишет 47 features. Это **23 секунды на трек**. Для 500 треков — 3 часа. При этом в финальный сет попадут ~15 треков. 97% работы впустую.

## Идея

Не все features нужны одновременно. DJ-процесс — это воронка:

```text
500 треков (импорт)
  ↓ "это вообще техно?"
400 треков (triage)
  ↓ "какой поджанр, какая энергия?"
 80 треков (плейлист)
  ↓ "из чего собрать сет?"
 50 кандидатов
  ↓ "какой порядок оптимален?"
 15 треков (сет)
  ↓ "как именно сводить?"
 15 треков (готовы к игре)
```

На каждом этапе нужны разные данные. Зачем считать beat grid для трека, который даже не техно?

## Четыре уровня анализа

### L1 — Triage: "Подходит ли трек?"

**Когда**: при классификации (`classify_mood`, `audit_playlist`)
**Что делаем**: скачиваем 30 секунд в RAM → 4 лёгких анализатора → удаляем файл
**Что получаем**: BPM, громкость (LUFS), энергия, спектральный профиль
**Что можем решить**:
- Это техно? (BPM 120-155, есть kick, не слишком гармоничное)
- Какой поджанр? (acid / dub / industrial / peak time / ...)
- Какой уровень энергии? (low / mid / high)
- В какой плейлист поджанра положить?

**Время**: ~5 секунд на трек (3с скачивание + 2с анализ)
**CPU**: ~60% одного ядра → можно 4-6 параллельных потоков
**MP3 на диске**: нет, удаляется сразу

### L2 — Placement: "Куда поставить в сете?"

**Когда**: при планировании сета (`build_set` — начальная расстановка)
**Что добавляется к L1**: тональность (key), MFCC (тембральный отпечаток)
**Что можем решить**:
- Расставить треки по шаблону (warm-up → peak → closing)
- Грубый порядок по энергии и BPM
- Группировка по тональности

**Время**: +1 секунда к L1 (key + mfcc на том же 30-сек клипе)
**Примечание**: L1 и L2 объединяем в один проход — нет смысла разделять, разница в 1 секунду

### L3 — Scoring: "Совместимы ли два трека?"

**Когда**: при оптимизации порядка (`build_set` — scoring, `score_transitions`)
**Что добавляется**: beat analysis (onset rate, kick prominence, HP ratio, pulse clarity)
**Что можем решить**:
- Полная 5-компонентная оценка перехода (BPM + harmonic + energy + spectral + groove)
- Greedy chain building с реальными scores
- GA-оптимизация порядка

**Время**: ~7 секунд на трек (3с скачивание + 4с beat analyzer на 60-сек клипе)
**CPU**: ~100% ядра → максимум 2-4 параллельных потока
**MP3 на диске**: нет, удаляется сразу
**Важно**: запускается только для треков-кандидатов в сет, не для всей библиотеки

### L4 — Transition: "Как именно сводить?"

**Когда**: при доставке сета (`deliver_set`)
**Что добавляется**: полный файл на диске (для проигрывания)
**Что можем решить**:
- Точки входа/выхода (секции: intro, drop, breakdown, outro)
- Cue points для быстрого доступа
- EQ-планирование

**Время**: ~3 секунды (только скачивание, файл сохраняется)
**MP3 на диске**: да — это финальные треки для проигрывания

## Как это связано с DJ-процессом

Основано на исследовании ISMIR 2013 (анализ реальных DJ-сетов) и руководствах DJ.Studio, DJ TechTools, SetFlow.

### Для классификации по поджанрам (L1)

Mood classifier использует 6 features: BPM, energy mean, LUFS, spectral centroid, HP ratio, kick prominence. Все доступны из L1. Исследование подтверждает: **энергия и тембр — главные различители** между поджанрами техно (ambient dub vs industrial отличаются по energy mean и HP ratio в 5-10 раз).

### Для расстановки в сете (L1 + L2)

DJ.Studio описывает конкретный пример 60-мин сета: BPM + LUFS + mood + key = достаточно для расстановки по шаблону (warm-up 124 BPM → peak 126+ BPM → release). Key нужен для планирования гармонии по Camelot Wheel.

### Для оценки совместимости (L3)

ISMIR paper показал: **тембральная похожесть (MFCC)** — самый значимый фактор в реальных DJ-сетах (p < 0.0001). Key, неожиданно, **не показал статистической значимости**. Профи подбирают треки по "звучанию", а не по тональности. Наша 5-компонентная формула (BPM 0.25 + Harmonic 0.20 + Energy 0.25 + Spectral 0.15 + Groove 0.15) может потребовать ребалансировки: поднять spectral, слегка понизить harmonic.

### Для перехода между треками (L4)

Beat positions и секции нужны только при реальном микшировании. Нет смысла считать beat grid для 500 треков, если играть будешь 15.

## Производительность

### Без параллелизации (sequential)

| Этап | Треков | Сек/трек | Итого |
|------|--------|----------|-------|
| Import (метаданные) | 500 | 0 | — |
| L1+L2 Classify | 500 | 5с | 42 мин |
| L3 Build set | 50 | 7с | 6 мин |
| L4 Deliver | 15 | 3с | 1 мин |
| **TOTAL** | | | **49 мин** |

### С параллелизацией

L1 анализаторы лёгкие (~60% CPU). Beat analyzer (L3) тяжёлый (~100%). Download не грузит CPU вообще. Это позволяет:
- L1+L2: **6 параллельных потоков** (CPU не bottleneck, сеть — bottleneck)
- L3: **4 потока** (beat analyzer тяжёлый, но на 60-сек клипе — терпимо)
- L4: **8 потоков** (только download)

| Этап | Треков | Потоков | Итого |
|------|--------|--------|-------|
| Import | 500 | — | — |
| L1+L2 Classify | 500 | 6 | **5 мин** |
| L3 Build set | 50 | 4 | **1 мин** |
| L4 Deliver | 15 | 8 | **6 сек** |
| **TOTAL** | | | **~6 мин** |

### Реалистичный сценарий (80 треков в плейлисте → 15 в сете)

| Этап | Треков | Время |
|------|--------|-------|
| L1+L2 Classify | 80 | **47 сек** |
| L3 Build set | 80 | **90 сек** |
| L4 Deliver | 15 | **6 сек** |
| **TOTAL** | | **~2.5 мин** |

### Сравнение с текущим подходом

| Подход | 500 треков | 80 треков |
|--------|-----------|-----------|
| Сейчас (всё сразу) | 3.2 часа | 31 мин |
| Tiered (sequential) | 49 мин | 10 мин |
| **Tiered (parallel)** | **6 мин** | **2.5 мин** |
| **Speedup** | **32x** | **12x** |

## Архитектура

### Уровни анализа в БД

```text
track_audio_features_computed
├── analysis_level: int (1, 2, 3)     ← НОВОЕ ПОЛЕ
├── bpm, lufs, energy_mean, ...       ← L1 features (всегда)
├── key_code, mfcc_vector, ...        ← L2 features (если level >= 2)
└── onset_rate, kick_prominence, ...  ← L3 features (если level >= 3)
```

### Pipeline по уровням

```text
L1+L2 (объединены):
  download_temp(30s) → [loudness, energy, spectral, bpm, key, mfcc] → delete_temp
  Выход: 35 features + mood classification

L3:
  download_temp(60s) → [beat analyzer] → delete_temp
  Выход: +12 features (onset, kick, hp_ratio, pulse, groove)

L4:
  download_permanent → [structure analyzer] → keep file
  Выход: MP3 файл + секции + cue points
```

### Когда что запускается

| MCP Tool | Уровень | Запуск анализа |
|----------|---------|---------------|
| `import_tracks` | — | Только метаданные, 0 анализа |
| `classify_mood` | L1+L2 | Авто для треков без features |
| `audit_playlist` | L1+L2 | Авто для треков без features |
| `distribute_to_subgenres` | L1+L2 | Авто для треков без features |
| `build_set` | L3 | Авто для треков с level < 3 |
| `score_transitions` | L3 | Авто для треков с level < 3 |
| `deliver_set` | L4 | Скачивание permanent |
| `analyze_track` | L3 | Явный запрос (как сейчас) |
| `analyze_batch` | L3 | Явный запрос (как сейчас) |

### Параллелизация

```text
                    ┌──────────┐
                    │ Downloader│ (1 поток, rate limit 1.5с)
                    │  Queue    │
                    └────┬─────┘
                         │ temp files
            ┌────────────┼────────────┐
            │            │            │
       ┌────▼────┐  ┌────▼────┐  ┌────▼────┐
       │Worker 1 │  │Worker 2 │  │Worker N │
       │ analyze │  │ analyze │  │ analyze │
       └────┬────┘  └────┬────┘  └────┬────┘
            │            │            │
            └────────────┼────────────┘
                         │ features
                    ┌────▼─────┐
                    │    DB    │
                    └──────────┘
```

1 downloader с очередью (rate limit = 1.5с между запросами)
N worker'ов анализа (N = 6 для L1, 4 для L3)
Download и analysis перекрываются: пока worker анализирует, downloader качает следующий

## Что меняется в коде

### Новое
- `analysis_level` поле в `track_audio_features_computed` + миграция
- Функция temp download (скачать → вернуть path → auto-delete)
- Parallel analyzer pool (asyncio + ProcessPoolExecutor)
- Level-aware pipeline: запускает только нужные анализаторы

### Изменения в существующих tools
- `classify_mood`: авто-запуск L1+L2 для треков без features
- `build_set`: авто-запуск L3 для кандидатов
- `deliver_set`: авто-download permanent для финальных треков
- `analyze_track`: по умолчанию L3, `--full` для L4

### Без изменений
- Все MCP tool интерфейсы (параметры, return types)
- Transition scoring формула
- Set templates
- Mood classifier
- DB schema (кроме нового поля)

## Выводы из исследования

Что подтвердилось из анализа профессиональных источников и научных статей:

1. **Тембр важнее тональности.** ISMIR 2013: MFCC-similarity — #1 фактор в реальных DJ-сетах. Key не показал статистической значимости. Наш spectral вес (0.15) может быть занижен.

2. **Energy ≠ BPM.** Профи говорят: "High energy ≠ fast BPM". Наши energy features (LUFS, energy mean) правильно отделены от BPM.

3. **Greedy ≠ оптимальный порядок.** ISMIR: nearest-neighbor хуже equal-step. GA-оптимизатор — правильный выбор для финального порядка.

4. **DJ-сеты ultra-precise по громкости.** Медиана скачка = 0.59 dB (vs 2.16 в альбомах). Наш порог 6 LUFS — generous, но корректен как hard reject.

5. **Percussion-only = key не важен.** Intro/outro без мелодий — key scoring можно ослабить. Это можно учесть в scoring если есть section data (L4).
