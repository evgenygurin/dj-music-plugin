# Карта репозиториев dj-music — с логикой решений

**Дата:** 2026-07-07 (переписано понятно). Здесь не просто список, а **почему**
каждая граница проведена именно так.

---

## 1. Правило: когда заводить отдельный репо

По умолчанию — НЕ дробить. Отдельный репо оправдан, если есть хотя бы одна причина:

1. **Разные тяжёлые зависимости** (librosa/numba/ffmpeg vs тонкий httpx).
2. **Разные секреты / граница доверия** (токен площадки vs пароль БД).
3. **Разный цикл релизов / владелец.**
4. **Разный способ запуска/масштабирования** (VM/GPU vs stdio-процесс).
5. **Переиспользование несколькими потребителями / самостоятельная ценность** —
   кусок нужен сам по себе, вне остальной системы.
6. **Граница состояния** (единственный, кто хранит данные, — отдельно).

Нет ни одной причины → это папка/пакет внутри существующего сервера, не репо.

## 2. Прогон по границам

### Площадки (Yandex/Beatport/SoundCloud/Suno и будущие) — КАЖДАЯ свой репо

Твоё решение, и оно попадает прямо в **причину №5**: каждый такой сервер должен
быть **самостоятельным продуктом** — установил только `dj-music-mcp-yandex` и
пользуешься Яндексом как отдельным MCP, без БД, gateway и всего остального.

Дополнительно работают и другие причины:
- №2 — у каждой площадки свои секреты и своя auth-специфика (Suno — session-cookie/
  bearer с истечением ~1ч; Yandex — OAuth-токен; Beatport — своё). Изоляция
  процессов = утечка одной не открывает другие.
- №3 — площадки живут своей жизнью: у Suno меняется web-API, у Beatport — каталог;
  чинишь одну, не пересобирая остальные.

Поэтому: **репо на площадку.** Каждый — stateless, зависит только от
`dj-music-contracts`, запускается и публикуется в одиночку. Новая площадка (Spotify,
Bandcamp, …) = ещё один такой репо по тому же шаблону. Это ядро системы —
расширяемый набор независимых интеграций.

Что важно для «пользоваться по отдельности»: каждый platform-сервер — полноценный
MCP со своими tools (`provider_read/write/search/download`), его можно подключить
напрямую в Claude Code / Cursor / любой MCP-клиент, минуя gateway и БД.

### Gateway — ОДИН, отдельного `platforms`-репо НЕ нужно

Агрегацию всех площадок делаем **внутри самого `dj-music-mcp`**, а не в отдельном
репо. Это штатный документированный паттерн FastMCP, а не костыль:

- `create_proxy({"mcpServers": {...}})` в одном сервере проксирует **несколько**
  бэкендов сразу, с авто-namespacing (`yandex_...`, `beatport_...`). Это и есть
  «промежуточные MCP всех площадок внутри одного gateway».
- Тот же сервер держит **свои** кросс-платформенные tools рядом с проксированными
  (доки прямо: «server has both local and proxied tools»; `LocalProvider`
  опрашивается первым → твой `unified_search`/дедуп живут рядом с площадками).
- Плата — прокси-латентность складывается по хопам; лечится плоской топологией
  (gateway → площадки напрямую, ≤2 хопа) + `ProxyProvider(cache_ttl=...)`.

Архитектурных проблем нет. Отдельный `dj-music-mcp-platforms` оправдан **только**
если «все площадки как один сервер» понадобятся как самостоятельный продукт для
внешнего потребителя вне DJ-системы (причина №5). Тебе не надо — **не заводим**.

Почему gateway всё же отдельно от площадок и БД: у него другая работа — он ничего
сам не делает, а **собирает** чужие серверы в одну поверхность и держит правила.
Ценность в том, что он тонкий и перенастраивается (добавить/убрать сервер, сменить
правила) **не трогая** серверы, которые делают реальную работу. На старте gateway
можно вообще отложить — роль «двери» временно берёт `dj-music-mcp-db`.

### БД — отдельный сервер (тут дробить правильно)

`dj-music-mcp-db` — причины №2 и №6: **единственный stateful** (пишет в Supabase),
свои креды, владелец identity трека (`tracks` — центр схемы, не размазывается по
площадкам). Сильное основание.

### Аудио и setbuilder — пока НЕ отдельные репо

- `dj-music-mcp-audio` — станет отдельным по причинам №1+№4 (тяжёлые депы + VM/GPU),
  но это **фаза 2**; пока пакет внутри `dj-music-mcp-db`.
- `dj-music-mcp-setbuilder` — чистый compute без особых причин; **пакет внутри БД**,
  выделять только если понадобится дёргать отдельно (№5).

### Контракты/протокол — НЕ пишем руками; это делает FastMCP

Первый инстинкт был завести `dj-music-contracts` с Python-протоколом `Provider` и
общими DTO — «чтобы все понимали форму данных». **Это лишний слой:** согласование
контрактов между серверами — штатная механика FastMCP, а не то, что делаешь сам.
Разберём по механизмам (все — в пределах пина `>=3.2.4,<3.4`).

**1. Композиция вместо ре-декларации (`mount` + `create_proxy`).** Gateway не
описывает и не реализует контракты площадок — он их **монтирует**. `create_proxy`
даёт живой мост поверх протокола MCP, `mount(..., namespace=...)` авто-неймспейсит
поверхность:

```python
main = FastMCP("dj-music-mcp")
main.mount(create_proxy("http://yandex-mcp/mcp"),   namespace="yandex")
main.mount(create_proxy("http://beatport-mcp/mcp"), namespace="beatport")
main.mount(create_proxy("./suno_server.py"),        namespace="suno")
main.mount(create_proxy("http://db-mcp/mcp"),       namespace="db")
# поверхность gateway: yandex_provider_search, beatport_..., db_entity_create, ...
```

Компоненты «зеркалируются» (mirrored) — отражают состояние удалённого сервера и не
могут молча разъехаться; чтобы изменить проксированный tool, делаешь `.copy()`.
Composition — код-шеринга ноль, интероп держит сам протокол.

**2. Схему ты не пишешь — она генерится из аннотаций, а клиент её регидратирует.**
Вот тот «более хороший вариант», который и смущал в подходе «согласуй схемы руками»:
JSON Schema (и input, и output) FastMCP **выводит из обычных Python type hints
(Pydantic)**. На стороне вызывающего `CallToolResult.data` **восстанавливает
полноценные типизированные объекты из output-схемы сервера** — datetime/UUID/
кастомные классы десериализуются автоматически. То есть «контракт» = твои же
аннотированные функции на одном конце и типизированные Python-объекты на другом.
Никакого IDL, `.proto` или ручного матчинга схем.

**3. Рассогласование формы (выход площадки ≠ вход db) — Tool Transformation, а не
общий DTO.** Единственное, чего composition не делает сам: «трек» площадки в её
форме vs `track_create` в форме db. FastMCP-native ответ — `ToolTransform` /
`ToolTransformConfig` / `ArgTransform` **декларативно** переименовывают, меняют
описание и **реформируют аргументы** прямо в точке монтирования на gateway:

```python
mount = main.mount(yandex_server, namespace="yandex")
mount.add_transform(ToolTransform({
    "yandex_provider_search": ToolTransformConfig(name="search_tracks", ...),
}))
```

Форму маппишь трансформом на границе — без общего пакета типов.

**4. …а в нашей системе чаще и трансформ не нужен: LLM — это клей.** Оркестрация уже
LLM-driven (`expand_playlist_workflow`: `provider_search` → прочитать результат →
`entity_create`). Claude сам перекладывает форму трека площадки в вызов db-tool'а →
статический контракт «площадка↔db» **emergent, живёт в prompt**. Плюс на большом
каталоге tools помогают `Tool Search` и экспериментальный `Code Mode` (заменяют
каталог на `search`+`execute` — LLM пишет код, чейнящий вызовы).

**Анти-дрейф контракта** — не общий класс, а встроенные tool fingerprinting
(`sha256(tool.key + to_mcp_tool())`), component versioning (`version=` + `VersionFilter`)
и `fastmcp inspect --format mcp` (посмотреть контракт).

**Итог:** Python-протокол `Provider` **убираем совсем** — это был механизм
in-process диспетчеризации монолита (`ProviderRegistry` + generic `provider_*`),
через границу процессов он не переносится. `dj-music-shared` **почти наверняка не
нужен**: заведи микро-пакет с DTO+константами (Camelot) только если появится
детерминированный (не-LLM) путь, где два+ сервера гоняют одну структуру и хочется
совпадения «по построению». Стартуй без него.

### Plugin — отдельный (другая задача)

`dj-music-plugin` — только манифесты хостов (Claude Code/Codex/Cursor) + витрина.
Это дистрибуция, не логика.

## 3. Итоговая структура

```text
dj-music-plugin              манифесты хостов → указывают на dj-music-mcp
      │
dj-music-mcp                 ЕДИНСТВЕННЫЙ gateway: собирает всё + правила + сценарии
   ├── dj-music-mcp-yandex        ┐
   ├── dj-music-mcp-beatport      │ самостоятельные площадки — каждая свой репо,
   ├── dj-music-mcp-soundcloud    │ работает и в одиночку, без остального
   ├── dj-music-mcp-suno          ┘
   └── dj-music-mcp-db            единственный stateful (Supabase, identity трека)
                                   └ пакеты audio/ setbuilder/ внутри — пока не выделены

   контракты держит FastMCP: mount+create_proxy (композиция), авто-схема из
   аннотаций + .data-регидрация (типы), ToolTransform / LLM-glue (форма данных)
   → своих протоколов/contracts НЕ пишем; dj-music-shared опционален
   агрегация всех площадок (unified_search/дедуп) живёт ВНУТРИ dj-music-mcp
```

**Чем FastMCP закрывает «согласование контрактов» (вместо ручного пакета):**

| Задача интеропа | Механизм FastMCP | Общий код? |
|---|---|---|
| Собрать площадки+db в одну поверхность | `mount()` + `create_proxy()`, авто-namespacing, mirrored components | нет |
| Описать вход/выход tool'а | авто-генерация JSON Schema из Python type hints (Pydantic) | нет |
| Получить типизированный результат у вызывающего | `CallToolResult.data` регидратирует объекты из output-схемы | нет |
| Форма «выход площадки → вход db» различается | `ToolTransform`/`ArgTransform` на gateway **или** LLM-glue в prompt | нет |
| Много tools на gateway | `Tool Search` / `Code Mode` (search+execute) | нет |
| Детект дрейфа контракта между релизами | tool fingerprinting + component versioning | нет |

| Репозиторий | Причина существования | Что внутри |
|---|---|---|
| **dj-music-shared** *(опц.)* | согласование формы данных, если нужно | только DTO (`CanonicalTrack`/`SearchResult`/`ExternalId`) + константы (Camelot). **Без Python-протокола** — контракт держит MCP tool-схема. Можно не заводить |
| **dj-music-mcp-yandex** | №5 самостоятельный продукт (+№2/№3) | адаптер YM, curated доменные tools, свои секреты |
| **dj-music-mcp-beatport** | №5 (+№2/№3) | адаптер Beatport (каталог/метаданные/матчинг) |
| **dj-music-mcp-soundcloud** | №5 (+№2/№3) | адаптер SoundCloud (писать с нуля) |
| **dj-music-mcp-suno** | №5 (+№2/№3) | генерация DJ-ассетов, session-auth |
| **dj-music-mcp-db** | №2+№6 единственный stateful | модели, репозитории, handlers, entity/compute/sync tools, resources, миграции; пакетами внутри — audio, setbuilder |
| **dj-music-mcp** | другая работа: композиция+правила, тонкий | `create_proxy`-конфиг (площадки+db), visibility/JWT/middleware, workflow-prompts / Code Mode. Можно отложить |
| **dj-music-plugin** | дистрибуция ≠ логика | `.claude-plugin/`, `.codex-plugin/`, marketplace; ссылка на gateway |

> Отдельного `dj-music-mcp-platforms` нет: агрегация всех площадок
> (`unified_search`, дедуп) живёт внутри `dj-music-mcp` через `create_proxy`
> над площадками + собственные local-tools. Отдельный репо — только если «все
> площадки одним сервером» понадобятся как продукт вне DJ-системы.

> **Про общий пакет:** Python-протокол `Provider` в распределённой схеме не нужен —
> контракты держит сам FastMCP (`mount`/`create_proxy` + авто-схема из аннотаций +
> `.data`-регидрация + `ToolTransform`; см. §2 и таблицу механизмов). Остаётся
> опциональный `dj-music-shared` только с DTO+константами, и стартовать можно без него.

**Отпочкуются позже, при конкретной причине:**
- `dj-music-mcp-audio` — когда аудио станет тяжёлым (№1+№4).
- `dj-music-mcp-setbuilder` — если понадобится отдельно от БД (№5).
- новые площадки (`dj-music-mcp-spotify`, …) — по шаблону площадки, каждая свой репо.
- `dj-music-mcp-template`, `dj-music-ops` — служебное, когда серверов много.

## 4. Как они взаимодействуют (простыми словами)

Роли:
- **общий язык** — не отдельный протокол, а **MCP tool-схемы, которые FastMCP
  генерит из аннотаций сам**: каждый сервер объявляет вход/выход своих tools,
  вызывающий получает типизированный `.data` (авто-регидрация из output-схемы).
  Gateway **не переописывает** площадки — он их `mount`+`create_proxy` (композиция);
  где форма трека площадки ≠ вход db, ставит `ToolTransform` на границе. Общий
  `dj-music-shared` (одни DTO) — опционально, только для детерминизма без LLM.
- **площадки (yandex/beatport/…)** — «руки наружу»: каждая ходит в свой сервис,
  отдаёт данные в общем формате, **ничего не запоминает**, держит только свои
  секреты. Работает как отдельный MCP сама по себе.
- **db** — «память»: единственный, кто пишет в Supabase, владелец id трека и связей
  трек↔площадка↔ассеты.
- **gateway (dj-music-mcp)** — «входная дверь»: снаружи один сервер, внутри — конфиг
  «проксируй площадки и db» + правила + многошаговые сценарии. Своей бизнес-логики нет.
- **plugin** — говорит Claude Code/Codex «вот один сервер, подключайся» → gateway.

Два способа пользоваться площадкой:
- **в одиночку** — подключаешь `dj-music-mcp-yandex` прямо в любой MCP-клиент,
  вызываешь его tools. Никакого gateway/БД не нужно.
- **в системе** — gateway проксирует ту же площадку и добавляет её в общую поверхность
  рядом с БД.

Пример потока «импортировать трек из Яндекса в библиотеку» (внутри системы):
1. Claude (через gateway) читает трек: `yandex.provider_read(...)` → canonical-DTO.
2. Claude пишет в библиотеку: `db.entity_create(track, ...)` → db резолвит/создаёт
   `tracks.id`, пишет `track_external_ids` + метаданные площадки.
3. Связку шагов задаёт workflow-prompt на gateway или Code Mode (LLM сам пишет код,
   чейнящий два вызова). Никакого «оркестратор-сервиса».

Ключевой принцип: **площадки не пишут в БД, БД не ходит в площадки** — каждый делает
своё, а gateway (или LLM) соединяет. Секреты площадки — только в её сервере, креды
БД — только в db, правила «кто кого дёргает» — только в gateway (по HTTP через JWT;
по stdio — через visibility/middleware). Это жёсткая инкапсуляция: каждый кусок
самодостаточен и используется как отдельно, так и в композиции.
